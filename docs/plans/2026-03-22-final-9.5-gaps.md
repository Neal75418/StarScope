## Task 1: Fix bare refetchInterval (2 places)

**Files:**
- Modify: `src/components/settings/DiagnosticsSection.tsx:35`
- Modify: `src/contexts/AppStatusContext.tsx:47`

**What to do:**

DiagnosticsSection — use useSmartInterval:
```typescript
import { useSmartInterval } from "../../hooks/useSmartInterval";

// In component:
const diagnosticsInterval = useSmartInterval(60_000);

// Replace:
refetchInterval: 60_000,
// With:
refetchInterval: diagnosticsInterval,
```

AppStatusContext — this is the health check. It SHOULD keep polling even when hidden (it needs to detect recovery). But it should be offline-aware. Create a simpler check:
```typescript
// AppStatusContext health check should still poll when hidden
// (to detect sidecar recovery), but stop when offline.
// Use a custom function instead of useSmartInterval:
refetchInterval: () => {
  if (!isOnline) return false;
  return 60_000;
},
```

**Commit:** `fix: apply smart polling to DiagnosticsSection, offline-aware health check`

## Task 3: Handle partial-failure — remove or implement

**Decision: Remove the type value.**

Rationale:
- `partial-failure` was defined speculatively but has no natural producer
- Dashboard's multi-query pattern already shows concatenated errors — adding partial-failure detection would require instrumenting every page's React Query results, which is over-engineering for a desktop app
- The existing 4 levels (online/offline/sidecar-down/rate-limited) cover all real-world scenarios
- YAGNI — remove it now, add it back when there's a real use case

**Files:**
- Modify: `src/contexts/AppStatusContext.tsx:17` — remove "partial-failure" from type
- Modify: `src/components/StatusBanner.tsx` — remove partial-failure icon
- Modify: `src/i18n/translations.ts` — remove partialFailure translation
- Modify: `src/App.css` — remove .status-banner--partial-failure
- Modify: `docs/engineering.md` — update degradation levels list

**Commit:** `refactor: remove unused partial-failure degradation level (YAGNI)`
